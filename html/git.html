Приклад реальної роботи з GitHub

Нам дали задачу зробити меню, ми це зробили і закомітили.

Перший рік, може більше, ми не будемо маржити в гілку мейн самостійно.

Як ми працюємо на командному проекті:

  • Ми запушуємо зміни в нашу гілку
  • Заходимо на репозиторій, де буде повідомлення, що наша гілка була тільки що запушена, і зелена кнопка Compare & pull request.
  • пулл реквест, це запит до тім-ліда, на те, щоб він перевірив наш код, і якщо він правильний, прийняв його в основну гілку.
  • Обираємо гілку, перевіряємо що конфліктів немає, з якоє в яку гілку ми хочемо помістити код, і натискаємо кнопку Create pull request.
  • Гарно його підписуємо.
  • Можно вибрати людей які будуть перевіряти мій шматок коду, натискаємо кнопку Create pull request.

Створення нової гілки, ...

git branch "branchName" - створює гілку з назвою "branchName".

git checkout "branchName" - переходить на гілку з назвою "branchName".

git checkout -b "branchName" - створює гілку з назвою "branchName" і переходить на неї.

Команди, щоб опублікувати свій footer на GitHub

$ git add .

To seal a commit, we must use the command:

$ git add commit -m "description of the commit"

git push - відправляємо на сайт github.

git push --set-upstream origin header або git push -u origin header - створюємо на github гілку і запушуємо код. Ця команда виконуєтся один раз після створення нової гілки. Надалі файли запушуються без цієї команди.

Далі натискаємо Compare & pull request (Створити пул реквест)

Якщо кнопки Створити пул реквест немаєє, заходимо у вкладку Pull requests / натискаємо кнопку New pull request / Обираю гілку

Далі операції які на проекті тімлід виконує самостійно

Як зайти в гілку яку створив колега, та подивитись що у нього.

git pull - отримуємо останні зміни з сайту github.

git checkout "branchName" - переходить на гілку з назвою "branchName".

git add та git commit, ми можемо записати однією командою

git commit -am "text commits" - переходить на гілку з назвою "branchName".

git push - відправляємо на сайт github.

Щоб зробити якісь дії в гілці, треба в ній знаходитись.

Якщо у нас, наприклад був відкритий файл index.html, і в гілці в яку ми переходимо теж є файл index.html, то у нас буде відкритий файл index.html тої гілки, в яку ми перейшли.

Як зміни які ми робили в своїй гілці зарелізити в основну гілку (опісля перевірки ментором).

Ми маємо знаходитись в тій гілці в яку ми хочемо зробити marge.

git checkout main - зливаємо гілку з назвою "branchName" в поточну гілку.

Перевіряємо, чи дійсно ми в гілці main

git branch

git marge "banchName" - зливаємо гілку з назвою "branchName" в поточну гілку.

У нас залишаєтся код в гілці main, і до нього додаєтся код який ми зробили.

Після marge нам необхідно сайт опублікувати в інтернеті.

Якщо ми забули, що нам теба робити, ми вводимо команду:

git status - видає підказку, що до наступної команди.

git push .

git marge "banchName" - зливаємо гілку з назвою "branchName" в поточну гілку.

Як видалити гілку.

Наприклад, ми замерчили з гілки, і вона нам більше не потрібна. Нам треба виділити дві гілки публічну і локальну.

  1. Треба знаходитись не в тій гілці яку ми збираємось видаляти.
  2. Видалити гілку з репозиторію на GitHub.

git branch -b "branchName" - видалення гілки з компьютера.

git branch --delete origin "branchName" - видалення гілки з репозиторію.

git push origin --delete "branchName" - видалення гілки з репозиторію другий спосіб.

Перед кожним marge робити перевірку командою git status. Всі зміни треба зберігти перед marge. (Робимо коміт та пуш).

Далі маємо знаходитись не тій гілці яку мерджім, і після цього merge потрібну гілку.

git push origin --delete "branchName" - видалення гілки з репозиторію другий спосіб.

Конфлікти merge та їх вирішення

Можно відкотити назад власний merge.

git merge --abbort - відміняємо merge (злиття гілок).

Два шматки коду, які виділені різними кольорами, зверху над конфліктуючими блоками 5 варіантів дій, з яких можно вибрати одну. Ось ці варіанти:

  1. Accept Change. Прийняти зміни.
  2. Accept Incoming Change. Прийняти вхідні зміни.
  3. Accept Both Changes. Прийняти обидві зміни.
  4. Compare Changes. Порівняти Зміни.
  5. Start Live Share Session. Почати сеанс прямого ефіру.
  • git status
  • git add . або git add file-name.ex;
  • git commit -am "text commits"
  • git push

Після того, як ми пофіксили конфлікт, нам треба ці зміни зафиксувати (додати add, підписати commit, відправити push на GidHub).

Потім видалити непотрібн гілку локально на компьютері, та на GitHub.

git branch -b "branchName" - видалення гілки з компьютера.

git branch --delete origin "branchName" - видалення гілки з репозиторію.

git push origin --delete "branchName" - видалення гілки з репозиторію другий спосіб.

git slash - зберігаємо не збережені зміни в файлах і кладемо їх в буфер обміну.

git stash apply - вставляємо збережені змни з буфера обміну. Зміни в буферу обміну зберігаются до того моменту поки проект зберігається на компьютері. Вставити збережене можна в будь-який файл, будь-якої гілки.

git diff - показує зміни, яки були внесені в код після останнього коміту. Поки є внизу дві крапочки, можемо скролити зміни за допомогою клави (стрілочки вверх - вниз).

Щоб вийти з режиму перегляду змін, треба натиснути клавішу "q".

Подивитись історію коммітів в якійсь branch

git log

Натискаємо "q" щоб вийти з цього, та взагалі будь якого режимую.

git restore --staged <file>...>

Commands for Git & GitHub

Initiation of Gid

Run the git init command in the project directory.

$ project_folder

$ git init

If we want to add a file to a commit:

$ git add <file>

If we want to add all files to a commit:

$ git add .

To seal a commit, we must use the command:

$ git add commit -m "description of the commit"

Synchronize a local repository with a remote one

$ git push repository_name branch

Creating a new branch

$ git branch new_branch_name

Switch to a new branch

$ git checkout new_branch_name

To create a new branch and switch to it immediately

$ git checkout -b new_branch_name

To move changes to the main branch:

$ git checkout main $ git marge new_branch

The merge conflict is usually resolved manually. We need to decide which code to keep and which to remove. After resolving the conflict, you need to reconvene the commit and synchronize the repository.

If you create a new task in a branch of the previous one.

Roll back the commit.

@ git revert

So that the commit disappears but all the changes we made remain. After that, the commit will disappear and the files will be marked as modified.

$ git reset --soft

If someone has made changes to the files of the branch to which we want to switch, Git will not allow it.

Hides all changes together

$ git stash

After saving the changes and switching to the new branch, we can apply the changes to the existing state.

$ git stash $ git checkout main $ git stash pop

Returns the local repository to the state to be specified.

$ git reset --hard
